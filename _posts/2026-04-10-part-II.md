# Part II — χ/na 2017–2018 & Secure Coding vs Code Security

## 2 — χ/na 2017–2018  
*Back‑in‑Time Engineering vs. US R&D — two timelines, two velocities*

2017–2018 was a strange moment in global tech:  
the US was talking about “AI revival”,  
while China was quietly building an R&D ecosystem that felt like it came from the future.

It was a period defined by:

- minimalist tools with surprising power  
- compression as acceleration  
- papers that didn’t fit the Western timeline  
- parallel evolution at different speeds  

Years later, when things like **DeepSeek‑V3** appeared,  
they felt less like breakthroughs and more like **continuations**  
of something that had already started back then.

This section is a checkpoint:  
a reminder that sometimes the future is visible long before it arrives.

## 3 — Secure Coding vs. Code Security  
*Two terms that look similar but live in different worlds*

### 3.1 — Secure Coding (OWASP): the craft of prevention

Secure Coding is about **writing code that resists being broken**.  
It’s craftsmanship, discipline, and habit.

It includes:

- input validation  
- sanitization and escaping  
- safe session handling  
- error handling without leakage  
- cryptographic hygiene  
- reducing attack surface  
- avoiding dangerous patterns  

### 3.2 — Code Security: the ecosystem around the code

Code Security is not about how you write code —  
it’s about **how code lives**.

It includes:

- static analysis (SAST)  
- dynamic analysis (DAST)  
- dependency scanning  
- secret scanning  
- SBOM generation  
- CI/CD security gates  
- policy enforcement  
- runtime protection  
- DevSecOps culture  

### 3.3 — The philosophical split

- Secure Coding → *“Make the code safe.”*  
- Code Security → *“Make the system safe even when the code changes.”*

Secure Coding is **grammar**.  
Code Security is **syntax + context + evolution**.

You need both.  
One without the other collapses.
